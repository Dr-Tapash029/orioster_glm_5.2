    17→    17→## 1. PRODUCT IDENTITY
    18→    18→
    19→    19→**ORIOSTER** IS AN **OFFLINE-FIRST, AI-ASSISTED, MULTI-TENANT HOSPITAL MANAGEMENT SYSTEM (HMS)** ENGINEERED
    20→    20→FOR LOW-RESOURCE HEALTHCARE ENVIRONMENTS. IT OPERATES AS A LOCAL-FIRST CLINICAL OPERATING SYSTEM WITH
    21→    21→ASYNCHRONOUS AI AUGMENTATION — NOT A CLOUD-DEPENDENT SAAS TOOL.
    22→    22→
    23→    23→> **CORE DESIGN PHILOSOPHY:**
    24→    24→> OFFLINE-FIRST · CACHE-FIRST · HUMAN-CONTROLLED · AI IS ASSISTIVE, NEVER AUTHORITATIVE.

    51→    51→## 4. IN-SCOPE FEATURES
    52→    52→
    53→    53→### CORE CLINICAL WORKFLOWS
    54→    54→- 10-STEP PATIENT ENTRY WIZARD (PATIENTENTRYWIZARD)
    55→    55→- VITALS RECORDING WITH LOCAL TRIAGE LOGIC (NO AI REQUIRED)
    56→    56→- CHIEF COMPLAINT CAPTURE (STRUCTURED DROPDOWN + OPTIONAL FREE-TEXT)
    57→    57→- PAST HISTORY TAGGING (STRUCTURED TAGS + CAPPED NOTES)
    58→    58→- ONGOING MEDICATION TRACKING WITH DUPLICATE AND INTERACTION WARNINGS
    59→    59→- HYPERSENSITIVITY / ALLERGY RECORDING WITH SEVERITY TAGGING
    60→    60→- APPOINTMENT SCHEDULING AND DOCTOR ASSIGNMENT WITH TIME SLOTS
    61→    61→- ROLE-BASED DASHBOARDS (DOCTOR / NURSE / ADMIN / LAB TECHNICIAN)
    62→    62→
    63→    63→### AI-ASSISTED MODULES (ORIO AI)
    64→    64→- DIFFERENTIAL DIAGNOSIS — 3 MOST-PROBABLE DIAGNOSES WITH CONFIDENCE METER
    65→    65→- TREATMENT PLAN GENERATION (POST DOCTOR-CONFIRMED DIAGNOSIS)
    66→    66→- PRESCRIPTION GENERATION (DOCTOR-CONFIRMED, CUSTOMIZABLE, EXPORT-READY)
    67→    67→- LAB REPORT ANALYSIS — PARAMETRIC ENTRY + AI CLINICAL FEEDBACK ADVISORY
    68→    68→- INVOICE GENERATION — AI-FILLED, FIXED TEMPLATE, ITEMIZED
    69→    69→- AI HUB — CENTRAL COMMAND MODULE FOR INVOICE, LABS, PRESCRIPTION, MEDICAL CERTIFICATE
    70→    70→
    71→    71→### INFRASTRUCTURE
    72→    72→- OFFLINE-FIRST SYNC VIA POWERSYNC SDK (BACKGROUND, INVISIBLE TO USER)
    73→    73→- ROLE-BASED ACCESS CONTROL (RBAC) ENFORCED AT RLS + ROUTER LEVEL
    74→    74→- END-TO-END ENCRYPTION (E2EE) ON ALL CLINICAL DATA VIA AES-GCM
    75→    75→- MULTI-TENANT ARCHITECTURE — DATA PARTITIONED PER FACILITY AND ROLE
    76→    76→- SQLCIPHER AES-256 ENCRYPTION AT REST

    93→    93→## 6. PRIVACY GUARANTEES (NON-NEGOTIABLE)
    94→    94→
    95→    95→1. **NO RAW PHI LEAVES THE DEVICE** — EVER, UNDER ANY CIRCUMSTANCE, FOR ANY REASON.
    96→    96→2. **AI RECEIVES ONLY `PATIENT_SUMMARY_V1`** — A DE-IDENTIFIED, COMPRESSED (≥70%) SUMMARY
    97→    97→   GENERATED FULLY LOCALLY AT STEP 8.
    98→    98→3. **AI CALLS ARE HARD-DISABLED UNTIL STEP 8 IS COMPLETE** — ENFORCED BY A CONTROLLER-LEVEL GUARD
    99→    99→   THAT CANNOT BE BYPASSED FROM ANY UI LAYER.
   100→   100→4. **ALL CLINICAL DATA IS AES-256 ENCRYPTED AT REST** (SQLCIPHER) AND AES-GCM ENCRYPTED
   101→   101→   BEFORE SYNC (E2EE VIA DART CRYPTOGRAPHY PACKAGE).
   102→   102→5. **AI LOGS ARE ENCRYPTED, SCOPED PER USER, AND PHI-SAFE** BEFORE WRITING TO ANY LOG SINK.

   157→   157→## 10. HERMES AGENT — ORCHESTRATION RULES
   158→   158→
   159→   159→THESE RULES GOVERN HOW HERMES AGENT MANAGES THE BUILD LIFECYCLE ACROSS SESSIONS.
   160→   160→
   161→   161→### 10.1 PRE-BUILD CHECKLIST (EVERY SESSION)
   162→   162→BEFORE WRITING ANY CODE, HERMES AGENT MUST:
   163→   163→1. INVOKE `AGENT-ORCHESTRATOR` → READ `PROGRESS_TRACKER.MD` TO IDENTIFY CURRENT KANBAN COLUMN
   164→   164→2. INVOKE `CODE-WORKFLOW` → READ `MEMORY.MD` IF IT EXISTS (PRIOR SESSION STATE)
   165→   165→3. READ `ARCHITECTURE.MMD`, `FOLDER_STRUCTURE.MD`, AND `CODE_STANDARDS.MD`
   166→   166→4. READ `UI_REGISTRY.MD` BEFORE WRITING ANY UI COMPONENT
   167→   167→5. READ `UI_TOKENS.MD` BEFORE WRITING ANY WIDGET CODE
   168→   168→6. ROUTE THE CURRENT TASK TO THE CORRECT PRIMARY SKILL PER §9 ABOVE
   169→   169→
   170→   170→### 10.2 DURING BUILD
   171→   171→- `KANBAN-WORKER` UPDATES THE TASK STATUS IN `PROGRESS_TRACKER.MD` AS WORK PROGRESSES
   172→   172→- `CLAUDE-CODE` WRITES CODE; `FLUTTER-DEV` ENFORCES PATTERNS; `CODE-WORKFLOW` MAINTAINS FILE ORDER
   173→   173→- `AGENT-ORCHESTRATOR` COORDINATES IF MULTIPLE DOMAINS ARE TOUCHED IN ONE TASK
   174→   174→- `SUBAGENT-DRIVEN-DEVELOPMENT` HANDLES TASKS THAT REQUIRE PARALLELISM ACROSS FEATURES
   175→   175→
   176→   176→### 10.3 AFTER EACH FEATURE
   177→   177→1. `TEST-DRIVEN-DEVELOPMENT` → WRITE REQUIRED WIDGET AND UNIT TESTS
   178→   178→2. `UI-UX-PRO-MAX` + `CLAUDE-DESIGN` → RUN `/IMPRINT` ON ANY NEW UI COMPONENT
   179→   179→3. `CODE-WORKFLOW` → RUN `/REVIEW` AGAINST `CODE_STANDARDS.MD` AND `ARCHITECTURE.MMD`
   180→   180→4. `KANBAN-ORCHESTRATOR` → MOVE TASK TO DONE COLUMN IN `PROGRESS_TRACKER.MD`
   181→   181→5. `CODE-WORKFLOW` → RUN `/REMEMBER SAVE` TO WRITE `MEMORY.MD`
   182→   182→
   183→   183→### 10.4 FAILURE HANDLING
   184→   184→- IF A BUILD FAILS, INVOKE `RECOVER` SKILL BEFORE CONTINUING
   185→   185→- IF SESSION CONTEXT IS POLLUTED, `KANBAN-ORCHESTRATOR` LOGS A RESET NOTE AND STARTS FRESH
   186→   186→- `AGENT-ORCHESTRATOR` PREVENTS CONCURRENT BUILDS THAT TOUCH THE SAME CONTROLLER OR SERVICE FILE
   187→   187→
